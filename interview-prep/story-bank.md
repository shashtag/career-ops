# Story Bank — Master STAR+R Stories

This file accumulates your best interview stories over time. Each evaluation (Block F) adds new stories here. Instead of memorizing 100 answers, maintain 5-10 deep stories that you can bend to answer almost any behavioral question.

## How it works

1. Every time `/career-ops oferta` generates Block F (Interview Plan), new STAR+R stories get appended here
2. Before your next interview, review this file — your stories are already organized by theme
3. The "Big Three" questions can be answered with stories from this bank:
   - "Tell me about yourself" → combine 2-3 stories into a narrative
   - "Tell me about your most impactful project" → pick your highest-impact story
   - "Tell me about a conflict you resolved" → find a story with a Reflection

### [Systems & Performance] Real-time Collaborative Canvas & CRDTs
**Source:** Report #009 — ProPro Productions — Founding Engineer
**S (Situation):** Designing a real-time collaborative infinite canvas whiteboard (pre-Excalidraw Figma-like solution) from scratch to support thousands of concurrent designers syncing drawings.
**T (Task):** Implement a high-performance synchronization runtime that handles real-time multi-user concurrency and connection drops on limited infrastructure resources.
**A (Action):** Initially, the team went with Operational Transforms (OT) for security since Google Docs used it. I realized OT was a poor architectural fit because it requires a 100% available, fault-tolerant central server, which our bootstrapped startup lacked. To overcome strong founder resistance due to prior heavy investment, I built a fast CRDT (Conflict-free Replicated Data Type) MVP to empirically demonstrate OT's severe failure modes under poor connection environments. This demonstration won buy-in to pivot. I migrated the runtime to decentralized CRDTs and optimized local clients to render thousands of collaborative elements at 60 FPS using QuadTree spatial partitioning, canvas virtualization, and DOM matrix transforms. To monetize early, I packaged the underlying core utilities (`UndoRedo` and `InfiniteCanvas` wrappers) as standalone modular packages and sold them separately under independent licensing.
**R (Result):** Pivoted to CRDTs with zero disruption, resulting in a **10% increase in customer satisfaction (CSAT)**. The core synchronization intellectual property and packages drove the company's successful technology-based acquisition.
**Reflection:** Designing collaborative architectures requires committing to rigorous mathematical data structures (like CRDTs) early. Rather than trying to debate architectural decisions theoretically, building focused, empirical MVPs that simulate failures is the most effective way to align stakeholders and overcome status-quo bias.
**Best for questions about:** System Design, High Performance, 0-1 Building, Startup Acquisition, Overcoming Resistance, MVPs, CRDTs vs OT.

### [Systems & Performance] High-Performance Data Ingestion & SQLite Keyset Pagination
**Source:** Project — Fiber AI — Independent Creator
**S (Situation):** A B2B prospecting dashboard needed to ingest, index, and query millions of company-to-technology references from a 20MB JSONL dataset. When scaling the database with standard offset-based pagination and word-match filters, response times degraded significantly, reaching over 2.5 seconds for large queries, violating typical API SLAs (<200ms).
**T (Task):** Build an O(1) memory ingestion pipeline and implement a database pagination and full-text search strategy to deliver sub-200ms query response times under heavy concurrent loads.
**A (Action):** Developed an ingestion pipeline utilizing Node.js streams and `JSONStream` to stream, clean, and map the JSONL data, writing bulk transaction chunks into `better-sqlite3` within O(1) memory. Shifted from SQL `LIKE` queries to SQLite FTS5 Virtual Tables, establishing database-level `AFTER INSERT/UPDATE/DELETE` triggers for instant index sync. Avoided limit/offset query performance degradation by designing a robust keyset (cursor-based) pagination query compiler using Kysely. Tested and validated query latency and bounds using Node's native test runner.
**R (Result):** Ingested millions of records with memory consumption capped under 45MB. Delivered database query response times consistently under 50ms (sub-200ms total API SLA) even at high offsets, and achieved zero lag in full-text index synchronization.
**Reflection:** I learned that relying on standard database offset pagination is a major scaling bottleneck for large tables because of linear scan degradation. By moving sorting and filtering constraints directly to indexed seek keysets (cursors), we can achieve O(1) query complexity regardless of table size.
**Best for questions about:** Database Optimization, Node Streams, SQLite/FTS5, Keyset Pagination, Low-Memory Pipelines, Performance Tuning.

**R (Result):** Successfully decoupled standard HTTP transactions from live telemetry. Handled thousands of concurrent WebSocket connections and live events with zero server thread blocking. Eliminated contract synchronization errors by consolidating Solana and Ethereum wallet handshakes.

### [Systems & Architecture] Sandbox-Safe DSL Formula Engine & Reactive Dependency Resolution
**Source:** Project — JoyFill Formula Engine — Advanced Systems Architect
**S (Situation):** Digital forms inside JoyFill required dynamic, Excel-like formula fields for real-time computations and conditional validations. However, existing standard practices relied on unsafe, XSS-prone client-side runtimes (like `eval()` or `new Function()`), or completely blocked the browser's single-threaded UI when resolving large, interdependent calculation chains with hundreds of calculated fields.
**T (Task):** Design a highly performant, 100% sandbox-safe calculation runtime and reactive dependency graph resolver. The engine had to guarantee sub-15ms latencies, absolute XSS security, and automated circular reference detection.
**A (Action):** Engineered a custom, sandbox-safe DSL evaluation runtime by parsing mathematical/logical formula text into Abstract Syntax Trees (ASTs) using `jsep`, and explicitly walking/mapping nodes to a modular formula library (`formulajs`). This completely decoupled string execution from browser globals (`window` or `document`), eliminating XSS injection vectors. Solved nested field dependency cascades (e.g., field `C = A + B`, `D = C * 0.1`) by modeling field relationships inside a Directed Acyclic Graph (DAG) and implementing Kahn's topological sorting algorithm in $O(V+E)$ time to compute evaluations in deterministic order and catch circular dependency loops during configuration. Finally, offloaded parsing loops and heavy statistical distributions to browser Web Workers via parallel task queues to keep the UI thread buttery smooth.
**R (Result):** Delivered calculations in under 15ms (10x faster than standard SLAs) with zero UI-thread stutter even on lower-end devices filling out massive forms. Secured a 100% XSS-safe execution environment and reduced downstream form validation support tickets by automating client-side arithmetic.
**Reflection:** I learned that building a custom parser and traversing an AST is not just an academic compiler exercise; it is an invaluable tool for enforcing execution security in high-scale client runtimes. Furthermore, applying graph algorithms (like DAGs) is the only clean, scalable way to manage state dependency resolution without introducing circular deadlocks or custom spaghetti code.
**Best for questions about:** DSL Compilers, Parsing, Graph Theory, Client-Side Performance, Web Workers, Sandbox Security, High-Performance Computations.

### [Algorithms & TDD] O(log N) Go Nearest-Slot Allocator with Min-Heaps
**Source:** Project — Go Nearest-Slot Parking Engine — Independent Creator
**S (Situation):** Required to build a high-performance, CLI parking lot allocator in Go that assigns incoming vehicles to the closest available spot (lowest-numbered slot ID first). Using naive sequential array scans would degrade to $O(N)$ operations as capacity grew.
**T (Task):** Guarantee highly performant, deterministic $O(\log N)$ or better scaling for allocations, retrievals, and state queries while maintaining absolute correctness.
**A (Action):** Built the state store around a binary Min-Heap (`availableSlots`) to keep track of available spaces, ensuring that popping the root always yields the lowest-numbered slot in $O(1)$ time and rebalancing takes $O(\log N)$. Coupled the heap with localized hash maps (`slotID -> Car`, `registrationNumber -> slotID`, and `color -> []slotID`) for $O(1)$ constant-time index lookups. Adopted a strict Test-Driven Development (TDD) cycle using AI-assisted pairing, writing failing tests first to establish behavior for invalid capacity inputs, case-sensitivity, double-parking, and duplicate registration plates before writing any implementation. Implemented thread-safe locks (`sync.Mutex`) to protect the shared heap from concurrency race conditions.
**R (Result):** Designed a bulletproof, 100% TDD-verified command line parking engine with $O(\log N)$ allocation/leave operations and $O(1)$ query speeds, handling edge cases gracefully with zero thread race conditions.
**Reflection:** Using the correct data structure (Min-Heap vs list scan) changes the fundamental scalability class of a system. Pairing this structural decision with rigorous TDD guarantees that performance optimizations do not introduce brittle regressions, especially when collaborating with AI agents to build tests first.
**Best for questions about:** Go, Binary Heaps, Data Structures, Concurrency Safe, TDD, AI-Assisted Workflows, Edge Cases.
